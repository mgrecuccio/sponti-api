Sponti - Notification Module Documentation

1. Overview

The Notification Module owns notification delivery for Sponti.

Its role is not to compute matching opportunities or decide matching business rules. Its role is to receive domain/application events, map them to notification types,
format notification commands, decide whether a user should be notified, persist real delivery attempts, dispatch push delivery, and retry transient delivery failures.

Current implementation supports:
- matching event listeners
- notification send decisions for cooldown and deduplication
- notification history with delivery status
- device token registration and disabling
- no-op push delivery for local/development mode
- FCM push delivery when Firebase is configured
- retry scheduling for transient push failures
- Firebase actuator verification

------------------------------------------------------------------------

2. Responsibilities

The notification module is responsible for:
- receiving application/domain events from other modules
- mapping events to notification types
- creating notification commands
- formatting notification title and body
- adding routing data for the mobile app
- applying send-decision rules before delivery
- preventing repeated generic suggestion notifications
- recording real notification delivery attempts
- storing and refreshing device tokens
- disabling invalid device tokens
- dispatching push notifications through the configured sender
- retrying transient provider failures
- exposing Firebase verification through an actuator endpoint

The notification module is not responsible for:
- computing match suggestions
- validating matching rules
- creating match proposals
- deciding availability overlaps
- owning contact or availability business logic

------------------------------------------------------------------------

3. Notification Types

Current notification types:
- MATCH_PROPOSAL_CREATED
- MATCH_SUGGESTIONS_AVAILABLE

3.1 MATCH_PROPOSAL_CREATED

Business meaning:
"Someone explicitly invited you to connect."

Triggered by:
- MatchProposalCreatedEvent

Target user:
- candidateUserId

Suggested mobile behavior:
- open or refresh incoming invitations
- call GET /api/v1/matches/incoming

This notification is concrete and should normally be sent once per stored proposal.

3.2 MATCH_SUGGESTIONS_AVAILABLE

Business meaning:
"There may be useful suggestions available now."

Triggered by:
- MatchSuggestionsAvailableEvent

Target user:
- userId from the event

Suggested mobile behavior:
- open or refresh suggestions area
- call GET /api/v1/matches/suggestions

This notification is generic and must be conservative because it is not a direct invitation from another person.
The matching opportunity scheduler may detect the same current availability opportunity repeatedly while the overlap is still active.
The notification module is responsible for suppressing repeated user alerts.

------------------------------------------------------------------------

4. Notification Command

Notifications are sent through SendNotificationCommand.

Command fields:
- userId
- type
- title
- body
- data

The data map contains routing and context information for the mobile app.

Example data for MATCH_PROPOSAL_CREATED:
- matchId
- initiatorUserId
- channelType
- targetScreen=incoming_matches

Example data for MATCH_SUGGESTIONS_AVAILABLE:
- targetScreen=match_suggestions
- candidateUserIds
- suggestionFingerprint
- bestScore
- overlapStart
- overlapEnd

The mobile app should not treat this data as source of truth. It should use it as a navigation hint and refresh from backend endpoints.

Current suggestion fingerprint:
- built from candidate user ids by MatchingNotificationListener

Recommended future improvement:
- include a normalized availability/opportunity key, for example candidate ids plus overlapStart and overlapEnd,
 so repeated suggestions for the same users in different availability windows can be distinguished more accurately.

------------------------------------------------------------------------

5. Event Listeners

5.1 MatchingNotificationListener

The matching notification listener belongs to the notification module.

It listens to matching events and maps them to notification commands.

Handled events:
- MatchProposalCreatedEvent
- MatchSuggestionsAvailableEvent

This preserves module boundaries:
- matching publishes business events
- notification decides how and whether to notify

For match suggestions, the listener adds:
- targetScreen=match_suggestions
- candidateUserIds
- suggestionFingerprint
- bestScore
- overlapStart
- overlapEnd

------------------------------------------------------------------------

6. Send Decision Logic

NotificationApplicationService applies send-decision logic before creating a delivery attempt.

6.1 General Flow

For every SendNotificationCommand:
- compute now from Clock
- call sendDecision(command, now)
- if the decision is suppressed, log the suppression and return
- if the decision is allowed, create a NotificationHistoryEntity
- save the history row
- dispatch delivery through NotificationDispatcher

Important behavior:
Suppressed notifications are not inserted into notification_history.

Reason:
Suppression for cooldown or duplicate suggestion detection is not a delivery attempt. Persisting every suppressed scheduler emission can create a high volume of low-value rows. The module logs suppressions but keeps notification_history focused on actual delivery attempts and their outcomes.

6.2 MATCH_PROPOSAL_CREATED

MATCH_PROPOSAL_CREATED currently bypasses generic cooldown and deduplication:
- sendDecision returns allowed immediately

Reason:
Each stored proposal is a concrete user action and should notify the candidate once.

Future delivery logic may still add idempotency by relatedMatchId to protect against duplicated events.

6.3 MATCH_SUGGESTIONS_AVAILABLE

MATCH_SUGGESTIONS_AVAILABLE uses conservative send-decision logic.

The service looks up only the latest successfully sent suggestion notification:
- same userId
- same notification type
- status SENT
- newest sentAt first

Statuses such as SUPPRESSED, FAILED, RETRY_PENDING, and NO_DEVICE_TOKEN are not used as cooldown anchors.

Reason:
Only an actual user-visible notification should influence future cooldown and deduplication. Suppressed or failed attempts must not extend cooldown.

6.4 Decision Outcomes

If no previous SENT MATCH_SUGGESTIONS_AVAILABLE notification exists:
- allow

If a previous SENT notification exists and cooldown has not expired:
- suppress with reason inside-cooldown

Cooldown check:
- latest.sentAt + sponti.notification.matching-suggestions-cooldown > now

If cooldown has expired and the new suggestion fingerprint is missing:
- allow

If cooldown has expired and the new suggestion fingerprint differs from the latest sent fingerprint:
- allow

If cooldown has expired and the fingerprint is the same:
- compare bestScore
- allow only if newBestScore > latestBestScore
- otherwise suppress with reason duplicate-or-not-improved

6.5 Same Users in Same Availability Slot

If the opportunity scheduler repeatedly finds the same current opportunity, for example:
- same userId
- same candidateUserIds
- same suggestionFingerprint
- same or lower bestScore
- still inside cooldown

then the notification is suppressed as inside-cooldown.

This is intentional. The matching scheduler is allowed to re-evaluate current opportunities periodically. The notification service is the boundary that prevents repeated user alerts for the same active opportunity.

7. Notification History

Notification history is stored in the notification_history table.

Current fields include:
- id
- user_id
- type
- related_user_id
- related_match_id
- sent_at
- metadata
- status
- provider
- provider_message_id
- attempt_count
- last_attempt_at
- next_retry_at
- failure_code
- failure_reason

Current statuses:
- PENDING
- SENT
- FAILED
- RETRY_PENDING
- SUPPRESSED
- NO_DEVICE_TOKEN

Current behavior:
- new allowed notifications start as PENDING
- dispatcher mutates the same history entity to SENT, NO_DEVICE_TOKEN, RETRY_PENDING, or FAILED
- cooldown/dedup suppressions are logged but not saved

SUPPRESSED still exists in the domain model, but the current matching-suggestion suppression flow no longer persists suppressed rows.

Business purpose:
- support cooldown decisions based on real sent notifications
- support retry decisions
- provide delivery observability
- avoid relying on in-memory state

Technical purpose:
- persist delivery state across application restarts
- support multiple backend instances better than in-memory tracking
- support retry selection through status and next_retry_at

------------------------------------------------------------------------

8. Device Tokens

Device tokens are stored in notification_device_tokens.

Device token registration endpoint:
- POST /api/v1/notifications/devices

Request fields:
- platform
- token
- deviceId
- appVersion

Delete/disable endpoint:
- DELETE /api/v1/notifications/devices

Delete request fields:
- token

Registration behavior:
- if the token already exists, refresh ownership, platform, device metadata, enabled flag, and timestamps
- if the token does not exist, insert a new enabled token

Deletion behavior:
- finds by authenticated user id and token
- disables the token instead of deleting it

Permanent provider failures also disable tokens during dispatch.

------------------------------------------------------------------------

9. Delivery Dispatch

NotificationDispatcher owns delivery state transitions.

Dispatch flow:
- if sponti.notification.push-enabled is false, mark history SENT with provider FCM and providerMessageId push-disabled:<timestamp>
- if push is enabled, load enabled tokens for the target user
- if no enabled tokens exist, mark history NO_DEVICE_TOKEN
- send one PushMessage per enabled token
- if any send succeeds, mark history SENT with provider message id
- if no sends succeed, select the primary failure and mark retry or failure

Permanent push failures:
- disable the device token
- mark the notification FAILED unless another token succeeds

Transient push failures:
- mark RETRY_PENDING if attempt count is below max attempts
- set next_retry_at using fixed delay multiplied by attempt count
- mark FAILED once max attempts are reached

Configuration failures:
- mark FAILED unless another token succeeds

------------------------------------------------------------------------

10. Push Senders

10.1 PushNotificationSender

PushNotificationSender abstracts provider delivery.

Input:
- PushMessage(token, type, title, body, data)

Output:
- PushDeliveryResult

Result fields:
- success
- providerMessageId
- failureCode
- failureReason
- failureType

Failure types:
- NONE
- PERMANENT
- TRANSIENT
- CONFIGURATION

10.2 NoOpPushNotificationSender

NoOpPushNotificationSender is active when:
- sponti.notification.fcm.enabled=false
- or the property is missing

Behavior:
- logs the push message
- returns success with providerMessageId noop:<timestamp>

This lets delivery flow be tested locally without Firebase credentials.

10.3 FcmPushNotificationSender

FcmPushNotificationSender is active when a FirebaseMessaging bean exists.

Behavior:
- builds a Firebase Message with token, notification title/body, command data, and type
- calls FirebaseMessaging.send
- maps successful provider ids to PushDeliveryResult.success
- maps FirebaseMessagingException to provider failure codes and failure types
- maps RuntimeException to FCM_RUNTIME_ERROR with CONFIGURATION failure type

Firebase messaging error mapping:
- UNREGISTERED -> PERMANENT
- SENDER_ID_MISMATCH -> PERMANENT
- INVALID_ARGUMENT -> PERMANENT
- UNAVAILABLE -> TRANSIENT
- INTERNAL -> TRANSIENT
- QUOTA_EXCEEDED -> TRANSIENT
- THIRD_PARTY_AUTH_ERROR -> CONFIGURATION

If Firebase has no MessagingErrorCode, the module falls back to Firebase ErrorCode and treats it as CONFIGURATION.

11. Retry Scheduler

NotificationRetryScheduler retries due transient delivery failures.

Schedule configuration:
- fixedDelayString = sponti.notification.retry.fixed-delay
- initialDelayString = sponti.notification.retry.fixed-delay

Selection query:
- top 50 histories
- status RETRY_PENDING
- next_retry_at <= now
- order by next_retry_at ascending

Retry behavior:
- if retry is disabled, return
- if attemptCount >= maxAttempts, mark FAILED
- otherwise rebuild SendNotificationCommand from history metadata
- dispatch through NotificationDispatcher

Important behavior:
The retry scheduler does not process SENT notifications and does not process suppressed notifications. It only queries RETRY_PENDING rows.

------------------------------------------------------------------------

12. Firebase Configuration and Verification

FirebaseConfiguration is active when:
- sponti.notification.fcm.enabled=true

Required property when FCM is enabled:
- sponti.notification.fcm.credentials-path

Beans created:
- FirebaseApp
- FirebaseMessaging

FirebaseActuatorEndpoint exposes Firebase verification through actuator endpoint id:
- firebase

Verification behavior:
- if FCM is disabled, returns UNKNOWN / SKIPPED with FCM_DISABLED
- if FirebaseMessaging is unavailable, returns UNKNOWN / SKIPPED
- if initialized, sends a dry-run topic message
- returns UP on successful dry run
- returns DOWN with failure details on Firebase exceptions

------------------------------------------------------------------------

13. Mobile Contract

Push notifications are signals, not state.

The mobile app should:
- receive push notification
- use targetScreen only as navigation hint
- call backend endpoint to refresh real data

For MATCH_PROPOSAL_CREATED:
- call GET /api/v1/matches/incoming

For MATCH_SUGGESTIONS_AVAILABLE:
- call GET /api/v1/matches/suggestions

The app should handle stale notifications gracefully because the suggestion or proposal may no longer be valid by the time the user opens the app.

14. Configuration

Current notification configuration:
- sponti.notification.matching-suggestions-cooldown
- sponti.notification.push-enabled
- sponti.notification.fcm.enabled
- sponti.notification.fcm.credentials-path
- sponti.notification.retry.enabled
- sponti.notification.retry.fixed-delay
- sponti.notification.retry.max-attempts

Current matching configuration that influences notifications:
- sponti.matching.opportunity-scheduler.enabled
- sponti.matching.opportunity-scheduler.fixed-delay
- sponti.matching.scoring.notification-minimum-score

Current defaults in application.yml:
- matching suggestions cooldown 30 minutes
- push-enabled false
- FCM enabled false
- retry enabled true
- retry fixed delay 5 minutes
- max retry attempts 3
- notification minimum score 60

Recommended MVP defaults:
- opportunity scheduler enabled
- fixed delay 5 minutes
- generic suggestion cooldown 30 minutes
- notification minimum score higher than normal suggestion minimum score
- FCM disabled locally unless credentials are available

15. Observability

Useful logs and metrics:
- notification commands received
- notifications allowed by send decision
- notifications suppressed by inside-cooldown
- notifications suppressed by duplicate-or-not-improved
- histories created
- histories marked SENT
- histories marked NO_DEVICE_TOKEN
- histories marked RETRY_PENDING
- histories marked FAILED
- delivery failures by failure type and failure code
- disabled tokens from permanent provider failures
- retry attempts and retry exhaustion

Important note:
Cooldown/dedup suppressions are currently log-only and do not produce history rows. If product analytics later needs suppression counts, add metrics or a separate aggregate table rather than writing every suppressed scheduler event to notification_history.

16. Production Considerations

16.1 Delivery Idempotency

Delivery should be idempotent enough to tolerate retries and duplicate events.

For proposal notifications, relatedMatchId can be used as a natural idempotency key.

For suggestion notifications, a stronger explicit notification fingerprint column would be better than parsing metadata.

16.2 Device Tokens

Device tokens can move between users or become invalid.

Current behavior refreshes existing tokens on registration and disables tokens on delete or permanent provider failure.

16.3 Multiple Backend Instances

Notification history is stored in the database, which helps across restarts and multiple instances.

However, if multiple scheduler instances publish the same suggestion event at the same time, stronger locking or unique delivery keys may be needed later.

16.4 Metadata Format

Metadata is currently stored as semicolon-delimited text.

This is sufficient for the MVP, but JSONB or explicit columns would be safer for:
- querying
- escaping values
- adding indexes
- storing structured delivery data

18. Summary

The Notification Module separates delivery concerns from business modules.

Matching decides when a matching event happens.

Notification decides how and whether to notify the user.

Only allowed delivery attempts are persisted to notification_history. Repeated generic suggestion events are suppressed by send-decision logic and logged without creating noisy history rows.
